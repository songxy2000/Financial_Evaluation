# 金融AI评测平台前端初版详细设计

## 一、文档目标

本文档基于现有《金融AI评测平台需求与设计》整理，目标是将业务概念稿进一步细化为一版可执行的前端设计方案，作为以下工作的共同依据：

- 官网与评测结果展示站点的页面设计
- 前端项目的信息架构与模块拆分
- 评测结果、产品资料、报名信息的数据结构设计
- 后续视觉设计、组件设计和开发排期的起点

本文档定位为 `V0.1 初版设计`，优先解决“先做什么、页面长什么样、信息如何组织、用户如何浏览与转化”四个问题，不追求一次性覆盖所有后台与运营能力。

## 二、设计原则

### 1. 产品定位

平台不是泛 AI 工具导航站，而是“金融 AI 产品评测与认证平台”。前端必须同时传达三层价值：

- 权威性：平台具备明确评测标准、专家复核机制和认证结果
- 专业性：聚焦金融行业，不做宽泛 AI 榜单
- 转化性：让参评机构愿意报名，让行业用户愿意查看报告与榜单

### 2. 设计原则

- 先建立可信度，再展示榜单内容，最后承接报名转化
- 首页讲清“为什么可信”，列表页讲清“谁更好”，详情页讲清“为什么得这个分”
- 数据表达优先清晰，不用花哨图表掩盖信息密度
- 桌面端优先，兼顾移动端浏览与报名
- 所有页面都要为后续接入后台发布系统预留结构

### 3. 初版范围

P0 前端页面建议覆盖以下内容：

- 首页
- 评测竞技场页
- 评测结果详情页
- 评测榜单页
- 报名参评页
- 关于我们页

## 三、目标用户与核心任务

### 1. 用户类型

前端设计需要兼容三类核心用户：

| 用户类型 | 典型角色 | 核心诉求 | 关键页面 |
| --- | --- | --- | --- |
| 参评方 | 金融 AI 产品负责人、市场负责人、售前负责人 | 了解评测价值、报名参与、查看认证权益 | 首页、奖项页、报名页、方法页 |
| 观察方 | 金融机构创新部门、采购方、业务负责人 | 快速筛选值得关注的金融 AI 产品 | 首页、列表页、详情页 |
| 行业传播方 | 媒体、研究机构、合作伙伴 | 获取榜单、奖项、平台方法论素材 | 首页、奖项页、关于我们页 |

### 2. 核心用户任务

| 用户任务 | 用户问题 | 前端承接方式 |
| --- | --- | --- |
| 了解平台 | 这是什么平台，靠不靠谱？ | 首页首屏、平台介绍、评测流程、合作背书 |
| 浏览榜单 | 哪些产品被评了，谁的表现更好？ | 列表页筛选、排序、标签、评分对比 |
| 深入查看 | 某个产品为什么得这个分？ | 详情页分项得分、适用场景、评语、认证状态 |
| 了解方法 | 评分怎么来的，是否专业？ | 方法页说明评分维度、流程、自动化与人工复核机制 |
| 申请参评 | 我如何报名，提交什么材料？ | 报名页表单、流程、所需资料、常见问题 |

## 五、视觉方向与品牌感

### 1. 关键词

建议前端整体风格围绕以下关键词展开：

- 专业
- 克制
- 权威
- 数据感
- 金融科技感

### 2. 视觉基调建议

- 以深蓝、石墨灰、银白为主色，突出金融与专业感
- 局部使用青绿色或金色作为“分数提升、认证通过、奖项荣誉”的强调色
- 首页背景不建议纯白平铺，可采用低对比网格、渐变光晕、金融数据线框等轻科技元素
- 字体建议中文使用更稳重的无衬线风格，数字分数和榜单编号可用更有识别度的数字字重

### 3. 页面气质要求

- 不能做成普通 SaaS 官网
- 不能过度炫技，避免“很酷但看不清数据”
- 首屏需要像行业机构平台，而不是个人作品集
- 评分、奖项、认证状态必须有明显视觉层级

## 七、页面详细设计

### 7.1 首页

### 页面目标

- 3 秒内让用户理解平台定位
- 证明评测体系专业可信
- 引导用户查看榜单或发起报名

### 页面模块建议

#### 1. 首屏 Hero

建议包含：

- 主标题：突出“金融 AI 评测与认证”
- 副标题：强调行业垂直、评测机制、榜单价值
- 主按钮：查看评测结果
- 次按钮：申请参评
- 右侧可视化：榜单卡片、评分雷达、认证徽章的组合视觉

建议文案方向：

- 标题示例：`面向金融行业的 AI 产品评测与认证平台`
- 副标题示例：`围绕业务能力、合规风险与用户适当性，建立可对比、可复核、可传播的金融 AI 评测标准`

#### 2. 平台价值区

使用 3 到 4 张卡片说明：

- 为什么做这个平台
- 对金融机构采购方的价值
- 对参评产品方的价值
- 对行业传播的价值

#### 3. 评测对象区

按类别展示：

- 金融大模型
- 金融智能体
- MCP / Skill / 金融工具接入能力

每类展示代表性机构 logo 或名称卡片，增强行业真实感。

#### 4. 评测方法概览区

建议用流程型模块说明：

1. 产品报名与资料提交
2. 自动化测试
3. 专家复核
4. 综合评分与认证
5. 榜单发布与年度奖项

此区域需要强烈传达“不是主观打分”的专业感。

#### 5. 评分体系概览区

拆成两个大卡：

- 业务能力评测
- 合规与风险评测

每个卡片展示评分项与权重，例如：

- 场景适配度 25%
- 业务准确率 25%
- 监管合规 25%
- 数据安全 25%

不建议首页一次性塞满全部说明，保留“查看完整方法”入口。

#### 6. 榜单预览区

展示最近发布的评测结果 Top N：

- 榜单排名
- 产品名
- 所属机构
- 综合得分
- 分类标签
- 奖项/认证标记

支持跳转详情页。

#### 7. 奖项与认证区

展示年度奖项名称与认证价值：

- 年度最佳金融AI产品
- 最佳风控AI
- 最佳客服AI
- 最佳营销AI
- 最具创新AI
- 年度新锐AI

配合“获奖带来的商业价值”说明，如品牌曝光、销售背书、市场传播。

#### 8. 报名转化区

独立成块，建议内容：

- 报名截止时间
- 可报名产品类型
- 标准服务与高级认证服务的说明
- 报名按钮

#### 9. 页脚信息区

包含：

- 平台简介
- 联系邮箱/微信/电话
- 法律声明
- 版权信息

### 首页关键交互

- 首屏按钮固定清晰，避免让用户滚动后才知道下一步
- 榜单卡片 hover 时可预览核心分数与标签
- 首页滚动过程中，评分体系和流程模块可加入轻量动效，但不要影响信息读取
- 移动端首屏按钮需要始终处于第一屏内

### 7.2 评测结果列表页

### 页面目标

- 支持用户快速筛选、比较、发现重点产品
- 将榜单浏览体验做得像“专业数据库”，而不是普通文章列表

### 页面结构

#### 1. 顶部页面头图

包含：

- 页面标题
- 当前评测年度
- 结果总数
- 简短说明文字

#### 2. 筛选栏

建议首版支持：

- 产品类别：大模型 / 智能体 / MCP / Skill
- 业务场景：风控 / 客服 / 投顾 / 研究 / 营销 / 通用
- 评测状态：已评测 / 评测中 / 即将发布
- 奖项标签：是否获奖
- 排序方式：综合得分 / 业务能力 / 合规风险 / 更新时间
- 搜索：产品名 / 机构名

桌面端可使用横向筛选条，移动端改为抽屉筛选。

#### 3. 结果列表区

每张结果卡建议包含：

- 排名或状态标签
- 产品 Logo
- 产品名 + 所属机构
- 一句话简介
- 综合得分
- 两个核心子分数
- 分类标签
- 奖项/认证标签
- 查看详情按钮

#### 4. 空状态设计

当筛选无结果时，应提供：

- 空状态提示
- 重置筛选按钮
- 推荐浏览分类

### 列表页交互建议

- 默认展示“综合得分排序”
- 切换排序时不刷新整页，只刷新结果区
- 筛选条件应在 URL 中可持久化，便于分享
- 卡片支持快速对比入口，后续可升级为多产品对比

### 7.3 评测结果详情页

### 页面目标

把“专业可信”真正落到内容细节上，这是整个平台最重要的内容页面。

### 页面结构

#### 1. 产品头部信息区

包含：

- 产品名称
- 所属机构
- 产品类别
- 当前状态
- 综合得分
- 奖项与认证
- 更新时间
- 官方链接入口

#### 2. 核心结论摘要区

用 3 到 5 条摘要快速说明：

- 适合什么金融场景
- 优势表现在哪些维度
- 哪些画像下表现更好
- 存在哪些风险或局限

#### 3. 评分总览区

建议展示：

- 综合得分大数字
- 业务能力分
- 合规风险分
- 可视化图表，可选雷达图或条形图

注意：

- 图表只是辅助，必须同时有清晰数字与文案说明

#### 4. 分项得分明细区

建议按两个大类分组展开：

- 业务能力评分项
- 合规与风险评分项

每个维度至少展示：

- 分值
- 权重
- 简要解释
- 评语
- 证据来源类型

#### 5. 用户画像适配区

这是平台差异化亮点，建议重点展示。

可用方式：

- 展示本产品测试覆盖的用户画像组合
- 标记“适配较好”“需要谨慎”“不建议”这类结论
- 用真实评测样例解释：保守型用户是否被错误推荐高风险产品

建议不要在初版中把 27 种画像全部展开成复杂矩阵，可先展示：

- 覆盖画像数量
- 典型画像案例 3 到 6 个
- 风险适当性总结

#### 6. 适用场景区

展示产品更适合的业务方向，例如：

- 智能客服
- 财富管理问答
- 风控辅助分析
- 研究投研助手

可采用标签加说明段落的组合。

#### 7. 风险与限制区

这一块必须保留，体现客观性。建议说明：

- 不适用的场景
- 已知局限
- 合规或解释性方面的注意事项

#### 8. 报告下载/咨询区

可提供：

- 查看完整报告按钮
- 商务咨询按钮
- 申请参评按钮

### 详情页重点要求

- 不能只是把分数摊开，必须有“结论摘要”
- 不能只展示优点，必须同时展示限制项
- 页面视觉层级要围绕“综合结论 -> 分项证据 -> 适用与风险”展开

### 7.4 奖项与认证页

### 页面目标

为平台商业化服务，突出获奖与认证的价值。

### 页面模块建议

- 奖项说明区：年度奖项类别及评选意义
- 获奖名单区：按年度展示获奖产品
- 认证说明区：认证等级、获得方式、使用权益
- 商业价值区：品牌传播、销售背书、市场曝光
- 申请入口区：引导报名

### 认证状态建议

可以设计为：

- 已通过评测
- 获得推荐认证
- 获得年度奖项

避免一开始设计过多等级，先保证用户能理解。

### 7.5 评测方法页

### 页面目标

解释“怎么评、谁来评、凭什么评”，增强可信度。

### 页面模块建议

- 方法总览
- 通用评测维度与权重
- 专项评测说明
- 用户画像系统说明
- 自动化评测与人工复核流程
- 评分结果生成机制
- 结果声明与免责声明

### 特别建议

- 原稿中的“风控AI / 客服AI 指标待定”应在页面上标记为“专项指标将逐步公开”
- 初版页面可先保留占位模块，避免结构断裂

### 7.6 报名页

### 页面目标

让参评机构清楚知道：为什么报名、怎么报名、需要准备什么。

### 页面结构

#### 1. 报名说明区

说明：

- 参评对象
- 报名价值
- 评测周期
- 报名流程

#### 2. 服务方案区

可先展示两类：

- 基础评测服务
- 高级认证服务

建议说明差异：

- 交付内容
- 周期
- 是否含认证/奖项评选资格

#### 3. 报名表单区

首版字段建议：

- 公司名称
- 产品名称
- 产品类型
- 核心业务场景
- 产品简介
- 联系人姓名
- 职位
- 手机号
- 邮箱
- 附件上传
- 是否同意条款

#### 4. 常见问题区

建议覆盖：

- 哪些产品可以报名
- 是否接受未正式发布产品
- 评测周期多长
- 是否提供详细报告
- 是否可用于市场传播

### 表单体验要求

- 支持分步填写优于单页超长表单
- 支持自动保存草稿是加分项，若初版时间不足可先不做
- 提交成功后需有明确成功页或成功态提示

### 7.7 关于我们页

### 页面目标

补足平台背景与对外合作信息。

### 页面模块建议

- 平台使命
- 发起背景
- 团队/专家机制
- 合作方式
- 联系方式
- 媒体与合作伙伴占位

### 文案方向

避免写成空泛公司介绍，重点强调：

- 为什么懂金融 AI
- 为什么有评测能力
- 为什么能做行业标准化工作

## 八、全站组件设计建议

### 1. 核心复用组件

建议优先建立以下组件：

- 顶部导航 Header
- 页脚 Footer
- HeroSection
- SectionTitle
- EvaluationCard
- ScoreBadge
- ScoreBar
- Tag
- AwardBadge
- CertificationBadge
- EmptyState
- FilterBar
- SearchInput
- StepTimeline
- FAQAccordion
- ApplicationForm

### 2. 组件设计原则

- 尽量模块化，首页卡片与列表卡片共用底层结构
- 评分组件要支持不同场景复用，例如大数字分、条形分、标签分
- 标签体系要统一，不同类别不要每页重新定义颜色

## 九、状态设计

前端初版经常被忽略，但建议一开始就明确以下状态。

### 1. 内容状态

- 加载中
- 无数据
- 发布中
- 已发布
- 即将发布
- 已下线

### 2. 交互状态

- 表单提交中
- 提交成功
- 提交失败
- 筛选中
- 搜索无结果

### 3. 标签状态建议

| 状态 | 颜色建议 | 含义 |
| --- | --- | --- |
| 已评测 | 深蓝/青绿 | 已发布结果 |
| 评测中 | 橙色 | 正在进行 |
| 即将发布 | 灰蓝 | 即将上线 |
| 获奖 | 金色 | 年度奖项 |
| 推荐认证 | 绿色 | 通过认证 |

## 十、响应式设计建议

### 1. 桌面端

- 以 12 栏网格为基础
- 首页与详情页采用宽屏信息布局
- 榜单与评分信息优先横向展开

### 2. 移动端

- 导航收起为抽屉菜单
- 首页模块改为纵向堆叠
- 列表页筛选改为底部抽屉或顶部按钮触发
- 详情页的评分图表简化为纵向条形图，避免过小雷达图难以阅读

### 3. 关键要求

- 移动端首屏必须保留核心 CTA
- 表单输入框尺寸、点击区域和键盘类型要适配移动端
- 表格类信息尽量卡片化，减少横向滚动

## 十一、内容策略建议

### 1. 首页文案节奏

建议按照以下顺序组织：

1. 平台是什么
2. 为什么可信
3. 评什么
4. 最近结果
5. 如何参评

### 2. 结果页内容风格

- 少口号，多结论
- 少“领先”“赋能”等泛词，多具体场景与评分依据
- 对外内容必须保持中性客观

### 3. 数据展示原则

- 数字要有单位和参照
- 分数旁尽量给出解释文字
- 排名与得分同时出现时，要避免用户误解为唯一绝对标准

## 十二、前端技术实现建议

如果这是一个全新前端项目，建议采用偏内容展示型官网架构，并兼顾后续后台接入。

### 1. 技术栈建议

- 框架：Next.js
- 语言：TypeScript
- 样式：Tailwind CSS 或 CSS Modules
- 内容管理：初版用本地 mock 数据，后续切换 CMS 或后台接口
- 图表：ECharts / Recharts
- 表单：React Hook Form + Zod

### 2. 数据接入策略

初版可分两阶段：

- 阶段一：使用本地 JSON / TS mock 数据完成页面结构
- 阶段二：接入后台发布系统和报名接口

## 十三、初版开发优先级建议

### Phase 1：可展示版本

- 完成首页
- 完成评测结果列表页
- 完成评测结果详情页
- 完成报名页
- 建立基础导航、页脚、卡片、标签、表单组件
- 使用 mock 数据打通全站

### Phase 2：可信度增强版本

- 增加评测方法页
- 增加奖项与认证页
- 完善关于我们页
- 增加更多评分说明与图表表现

### Phase 3：业务可运营版本

- 接入后台发布
- 接入报名提交接口
- 增加 SEO 配置
- 增加埋点和转化监测

## 十四、当前待确认问题

以下问题不影响初版前端结构设计，但建议尽快确认，以减少后续返工：

- 奖项是否按年度区分，是否需要历史届次展示
- 认证是否分等级，还是只做“通过/推荐/获奖”三级表达
- 评测结果是否允许公开全部详细分项，还是仅展示摘要版
- 报名是否有付费流程，若有，是否需要支付或商务预约节点
- 首页是否需要展示合作机构、专家委员会或媒体背书
- 专项评测维度何时补全，尤其是风控 AI 与客服 AI

## 十五、结论

前端初版的核心不是“把几个页面做出来”，而是先把平台的可信结构搭起来。建议围绕“首页建立认知、列表页体现价值、详情页展示专业、报名页完成转化”的最小闭环推进，这样即使第一版内容仍在完善，也能具备对外展示与商务沟通的能力。

如果下一步进入页面实现，建议优先从首页、评测结果列表页和评测详情页开始，因为这三页将决定平台是否看起来像一个真正可运营的金融 AI 评测平台。
